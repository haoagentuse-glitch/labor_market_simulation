## 1. 專案一頁地圖（架構回顧）

### 設計哲學（三條，務必守住）

1. **減法先於加法**：模型砍到「三純量 + 一條規則」的最小骨架。能刪的都刪了。
2. **浮現，不是設定**：有趣的結果（替代、集中、掏空）都要從規則**長出來**，不能寫進公式。任何把結論直接塞進方程式的設計都算退步。
3. **一次只加一個回饋**：要擴充時，一步一個機制，各自留下相圖，才分得清誰造成什麼。

### 三純量一規則（速記）

```
σ 任務難度（離 AI 前沿多遠）   a 人的能力   F AI 前沿（以速度 r 上爬）
規則：σ<F → AI 替代；σ≥F → 需 a≥σ 的人協作，報酬 V(σ)=σ^k；排序匹配（強配難）
已加回饋：在崗者學習 g、失業者再訓練 retrain_rate
```

## 2. 現況：第一輪（L_0.1.0）跑出了什麼

四個核心結論（細節見研究回顧）：

1. **前沿速度不對稱地主宰命運**：崩潰/重配置的邊界接近一條**垂直臨界線**（r≈0.004），不是 r=g 對角線。學習 g 能把線往右推一點，但**前沿太快時救不回**。
2. **任務生成 ≠ 重新就業**：新工作生成率（→0.8）遠超 AI 替代率（→0.3），失業卻仍從 25% 升到 42%——新工作都在高端，被替代者搆不到。
3. **掏空＋升級＋失業同時發生**，由「技能能否跟上前沿」這條主軸驅動。
4. **集中度指標被就業水準汙染（confound）**：分配相圖反直覺地顯示「能力越均質越集中」，但那其實是「倖存者稀少」造成的假象，不是真的超級明星。

## 3. 重點：哪些是承重牆、哪些已知脆弱

### 承重牆（動之前要非常小心，會改變一切）

- 三純量一規則、排序匹配的邏輯。
- σ/a/F 的尺度校準（能力中位錨在 0.5、F 掃 0.2→0.9）——動尺度會讓所有結果不可比。
- r-vs-g 的賽跑框架（這是目前最穩的結論）。

### 已知脆弱／待修（這個 task 的素材）

| 問題 | 後果 | 修法難度 |
|---|---|---|
| **集中度指標 confound** | 分配相圖結論不可信 | 低（加「在職者 Gini」分開看） |
| **過程圖只跑單一 seed** | 可能是特例 | 低（多 seed 中位數+分位帶） |
| **能力只升不降** | 高估長期適應力 | 中（加技能折舊） |
| **報酬無分租規則** | 分不清不均在勞工或資本 | 中（加 worker 分潤份額） |
| **前沿外生（F 線性）** | 看不到臨界點/路徑依賴 | 中（learning-by-doing） |
| **無需求面** | 看不到連鎖崩潰/良性循環 | 高（就業→所得→需求迴路） |

## 4. 這個 Task 的核心抉擇：A 還是 B

> 你提出的問題：**直接增加參數/變因（A）**，還是**完善基礎版、再跑一次基礎驗證（B）**？

### Option A：直接加參數/變因
加內生前沿、需求面、分租等新機制，往更真實、更豐富走。

- **好處**：更接近現實，可能更快看到「意想不到的」湧現（臨界點、連鎖崩潰）。
- **代價/風險**：**在指標還被汙染、過程還沒多種子確認時加機制，等於在一把歪掉的尺上蓋樓**——新結果分不清是新機制造成的，還是舊 bug 放大的。違反「一次只加一個回饋」的前提（前提是底層是乾淨的）。

### Option B：完善基礎版 + 重跑基礎驗證
先把已知脆弱點修掉，重跑一次乾淨的基準，確立可信的 baseline，再談擴充。

- **好處**：把第一輪那個 confound 結論**證實或推翻**；建立一個之後所有實驗都能對照的乾淨基準；成本低、見效快。
- **代價/風險**：短期內不會有「新現象」，感覺像在原地打磨。

## 5. 我的建議：先 B，後 A（但 B 要小而快）

理由很簡單：**第一輪已經有一個結論被證明是量測假象（集中度 confound）。在這個還沒解決前加新變因，會讓你無法判斷任何新結果的真假。** 科學上，先確保「尺是準的」再「蓋樓」。

而且 B 很便宜——下面這份最小清單大概一個工作段就能做完，做完你會得到一個**可信、可對照的 baseline**，之後每加一個機制都能乾淨地歸因。

### B 的最小清單（建議這個 task 的主體）

1. **拆開集中度指標**（最高優先）：在 `model.py` 加「**在職者 Gini**」（只算 earnings>0 的人）與保留「就業率」兩條獨立指標；把 `phase_distribution` 重畫一次。→ 直接驗證或推翻「能力越均質越集中」這個反直覺結果。
2. **過程圖多種子**：`process.py` 改成跑 N 個 seed，畫中位數 + 分位帶，確認掏空/升級/錯配不是特例。
3. **量化臨界前沿速度 r\***：沿 g 掃描，找每個 g 對應的崩潰臨界 r，畫成 `r*(g)` 曲線——把「前沿有結構性優勢」變成一條可引用的線。
4.（可選）**最小敏感度檢查**：對核心尺度參數（F0、ability 中位、task_density）各 ±20%，確認結論方向不變（穩健性）。

做完 1–3，更新一份 `docs/implementation/RESEARCH_REVIEW_v0.2.0.md`，再決定 A 要先加哪個機制（我的順序建議：**內生前沿 → 需求面 → 分租**）。
