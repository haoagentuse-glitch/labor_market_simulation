# 研究型專案文件規範(SOP)

> 類別:sop | 狀態:active | 日期:2026-07-03
> Canonical 版本是個人層 skill `~/.claude/skills/research-docs/SKILL.md`;本檔為專案副本,內容一致,更新時兩邊同步。

## 1. 三條憲法

1. **目錄扁平**:所有文件在 `docs/` 同一層,禁止子目錄。歸類與排序全部交給檔名。
2. **檔名即資料庫**:檔名 = `類別碼_類別名_定位符_slug`,不打開檔案就知道性質、版本、新舊。
3. **單一事實來源**:每項資訊只有一個家;第二次出現必須是連結。兩份文件講同一件事 = 歸類錯誤,當場合併或重切。

## 2. 命名文法(違規檔名不得存在)

**Singleton 類**(每專案恰好一檔,無定位符):

```
00_index.md   01_sop.md   05_evolution.md   50_sparks.md
```

**Collection 類**(定位符必填,缺定位符 = 違規):

```
10_concept_v<N>[_<slug>].md          思想迭代(版本)
20_spec_v<N>[_<slug>].md             技術迭代(版本)
30_exp_ledger.md                     實驗台帳(每類唯一保留字 ledger)
30_exp_<實驗版號>[_<slug>].md         單版實驗檢討(如 30_exp_L0.2.0.md)
40_log_<YYYYMMDD>_<slug>.md          討論/決策紀錄(日期)
90_raw_<YYYYMMDD>_<slug>.md          凍結素材(歸檔日期)
```

- slug 一律 kebab-case、≤4 個詞。
- concept 與 spec 的版號各自獨立計數;若歷史沿革導致起始號不同(如本專案 spec 從 v4 起算),在 `00_index` 註明一次。

## 3. 通用硬規則(反 slop)

1. **檔頭資訊列**:每份文件標題後第一行固定為
   `> 類別:X | 狀態:active / frozen / superseded(由 Y 取代) | 日期:YYYY-MM-DD`
2. **列點必須承重**:每個列點至少含一個具體指涉(數字、檔名、參數名、機制名、文獻名)。寫不出具體指涉的列點直接刪。
3. **禁止項**:重述標題的開場段;空洞強調詞(「非常重要」「值得注意的是」——重要性用位置表達,放前面);沒有內容的展望句(「未來可以進一步探索」);同一資訊第二次以全文出現。
4. **字數上限**(不含表格與程式碼;超限 = 還沒想清楚,回去砍):
   concept 3000 / spec 3000 / exp 2500 / log 1500 / sparks 每則 150。
5. **凍結紀律**:frozen 文件唯一可動處是檔頭狀態列與文末「後記(日期)」區塊;正文永不改。改寫需求一律走「新版本檔案 + 原稿移入 90_raw」。

## 4. 各類文件強制模板

節序固定、不增不減(標「可選」者除外)。每節後括號是該節的寫作邏輯。

### 00_index(地圖)

```
# 00_index — 文件地圖與進度
1. 文件地圖    (表格:檔名連結|類別|狀態|一句話。一份文件一列,沒進表=遺失)
2. 目前進度    (≤6 個列點,每點含數字或檔名;只寫「已成立的事實」)
3. 下一步      (編號 ≤3 項,每項可直接動工;有分岔就寫分岔條件)
```

### 05_evolution(敘事總覽)

```
1. 故事版      (30 秒、零術語,幕次結構:第一幕/第二幕/…)
2. 一句話定位  (整個研究在做什麼,一句)
3. 時間線      (表格,一階段一列,欄位=各軌道;只放里程碑)
4. 分軌指引    (每軌一行:看什麼去哪份文件)
5. 對外定位    (論文/比賽角度主打什麼;進度不寫這裡,指向 00_index)
```

### 10_concept(思想迭代)

```
0. 與前版差異速查  (表格:前版|本版|為什麼。v1 免)
1. 核心提問        (一段,以問號結尾;回答「在問什麼、為什麼值得問」)
2. 立場與機制      (本版的主張,一主張一小節;每個主張講清楚「它取代了什麼舊想法」)
3. 可證偽假設      (H1…Hn,每條必附「什麼觀察會推翻它」;寫不出推翻條件的不是假設,刪)
4. 未決問題        (編號;每條寫清楚「什麼證據能了結它」)
5. 交棒            (本版把什麼交給 spec / exp / 下一版 concept)
```

### 20_spec(技術真理來源)

```
0. 與前版差異速查  (表格。v1 免)
1. 系統一頁        (狀態變數表 + 全部規則,放得進一個畫面;讀完可以動手實作)
2. 參數表          (名|意義|基線值|角色:核心/掃描/地基)
3. 機制註記        (每機制一小節:規則、為何是最小形式、它讓什麼「免費浮現」;
                    設計中途的重大修正保留為小節,註明「實作否證了什麼」)
4. 輸出指標        (結果指標與過程指標;體制判定的客觀定義,不靠眼睛看)
5. 待加清單        (排序;一次只加一個,每項寫預期會多出什麼現象)
6. 工程            (環境、目錄、驗證方式,≤10 行)
規則:spec 不寫研究動機(連 concept)、不寫實驗結果(連 exp)。
```

### 30_exp(單版實驗檢討)——由淺入深,故事先行

```
檔頭補充:版本 | git hash | 產物路徑 results/<版本>/
0. 先講個故事      (≤5 句、零術語;外行人讀完能轉述)
1. 一張圖 + 摘要   (最關鍵一張圖;headline ≤3 條,每條含數字)
2. 設定            (參數、種子、與上一版的差異;可重現所需的一切)
3. 逐項深講        (一結論一小節:證據→機制→與假設對照)
4. 邊界與誠實      (侷限、意外、疑點;疑點必須標出,吞掉疑點=造假)
5. 下一步          (≤3 項,依投入產出排序,每項可直接動工)
驗收:讀者停在任何一節,都得到一個完整但深度不同的理解。
負結果照此模板寫,照樣編號。
```

### 30_exp_ledger(實驗台帳)

```
開頭:結果資料歸檔規範(見 §5)的專案級實作說明(版本字串來源、目錄約定)
每版一節、新在上,節內固定五欄:
  狀態(✅/待跑) / 動機(一句) / 改了什麼(編號) / Headline(含數字) / 產物與檢討連結
舊節永不改。
```

### 40_log(討論/決策紀錄)

```
1. 問題    (當時要解決什麼,一段)
2. 脈絡    (討論實際怎麼走,依時序;保留轉折,不美化成直線)
3. 結論與決策  (編號;每條標【已決】或【待決】)
4. 產出    (交給哪份文件:進 concept 新版?進 spec 待加清單?)
寫完即凍結。
```

### 50_sparks(靈光筆記)

```
每則固定三段:卡在哪 → 想通了什麼 → 為什麼關鍵。
一則 ≤150 字;append-only,新則加在最後;可有一節「暗線」總結共同模式。
```

### 90_raw(凍結素材)

```
唯一要求:檔案最頂端加一個引用塊,寫明「這是什麼、為何歸檔、被哪份文件取代/消化」。
其餘原樣保留,檔內連結允許失效。
```

## 5. 結果資料歸檔(可重現性)

實驗產物不進 `docs/`,凍結在 `results/<版本>/`:

```
results/<版本>/
├── figures/   *.png      可由腳本重生,git 忽略
├── data/      *.csv      每張圖的文字版陣列,進版控
├── summary.json          該版所有純量結論(不看圖能讀結果)
└── manifest.json         參數/種子/git hash/時間
```

無 manifest = 不可重現 = 視同不存在。版本語意:加機制或改結論 → minor;重跑或修圖 → patch。

## 6. 寫作時機(什麼事件觸發哪份文件)

| 事件 | 動作 |
|---|---|
| 想法/提問變了 | `10_concept_v(N+1)`,附差異速查表 |
| 設計/規則變了 | `20_spec_v(N+1)`;小修在現版內並於差異表補一列 |
| 跑完一輪實驗 | `30_exp_ledger` 加一節 + 獨立 `30_exp_<版號>` 檢討 |
| 重要討論或決策 | `40_log_<日期>_<slug>`,當天寫,寫完凍結 |
| 想通了一個轉折 | `50_sparks` 追加一則 |
| 文件被取代/來源素材 | 移入 `90_raw_<日期>_<slug>`,頂端加歸檔說明 |
| 任何以上動作之後 | 更新 `00_index`(地圖一行 + 進度) |

## 7. 新專案啟動 checklist

1. 建 `docs/`,複製本規範為 `docs/01_sop.md`。
2. 建 `docs/00_index.md`(三節骨架)。
3. 寫 `docs/10_concept_v1.md`(照模板,含可證偽假設)。
4. 有實作時開 `docs/20_spec_v1.md` 與 `docs/30_exp_ledger.md`。
5. 第一輪結果 → 第一份 `30_exp_*`,嚴格由淺入深。

## 8. 交付前 checklist(每份文件出手前自查)

- [ ] 檔名符合 §2 文法(定位符齊全)?
- [ ] 檔頭資訊列存在且狀態正確?
- [ ] 節序與所屬模板完全一致?
- [ ] 每個列點都有具體指涉?
- [ ] 沒有任何資訊是第二次全文出現?
- [ ] 字數在上限內?
- [ ] `00_index` 已同步?
